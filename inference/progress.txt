Discovery Strategy Chosen
- EIS discovery:
  - Filesystem-based.
  - Scan /run/user/<uid>/gamescope-*-ei.
- Gamescope socket discovery:
  - Filesystem-based.
  - Scan /run/user/<uid>/gamescope-* and exclude -ei and .lock files.
- PipeWire node discovery:
  - Host-command-based.
  - Use pw-cli ls Node and wpctl inspect <id>.
  - Filter for node.name = gamescope and media.class = Video/Source.
  - Resolve the client id and inspect the client for uid/process metadata.
- Multi-instance safety approach:
  - Snapshot sockets and node ids before launching a slot.
  - Launch the slot.
  - Poll for newly added EIS socket plus newly added gamescope Video/Source node for that user.
  - Persist the resulting mapping in worker state.

Architecture Chosen
- Best-fit architecture selected:
  - One host supervisor process for API/UI/control.
  - One isolated worker object/process boundary per slot.
  - One config file for slot definitions and runtime defaults.
- Supervisor responsibilities:
  - Load config.
  - Manage slot workers.
  - Expose control API and browser UI.
  - Build a composite inspection image.
  - Later own recording and model-facing transport coordination.
- Worker responsibilities:
  - Launch tmux session for its slot.
  - Discover gamescope socket, EIS socket, PipeWire node.
  - Start capture/injection bridge.
  - Track slot state, errors, and latest frame.
  - Accept input events for injection.
